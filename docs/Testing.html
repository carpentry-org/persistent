Testing

How persistent is tested.

Approach

Tests are deterministic and semantic. Each collection has its own suite under test/. Heavy randomized fuzzing lives in rc, because that's where the shared-ownership lifetime risk concentrates. Verifying rc holds plus checking that each collection wires it up correctly is the contract here.

What's covered for every collection

  • constructors and emptiness
  • the structure-specific updates (insert/update/remove/push/pop/get)
  • persistence: older versions remain unchanged after updates
  • ptr-eq behavior on branching
  • iteration (reduce, each, to-array)
  • structural = where defined
  • per-suite memory balance (no leaks, no double-frees)
  • deep-input regressions (e.g. list drop-n at 200k, others at 100k)

A few structure-specific checks: trie prefix lookup, ordered map min/max and successor-based delete, heap sorted-pop progression, hash map Int.MIN hash-path regression, vector assoc and pop-back branch behavior.

Running

Single suite:

carp -x test/persistent_list.carp --log-memory --no-profile

Full validation matrix (sanitizers + multiple optimization levels):

./scripts/validate.sh

Lanes: O1 with ASan/UBSan; O2 release-like; O3 optimizer stress (PERSISTENT_VALIDATE_RUN_O3=0 to skip).

Knobs

  • PERSISTENT_OPT_LEVEL: O0|O1|O2|O3 (default O1)
  • PERSISTENT_SANITIZE: 1|true|yes enables sanitizers
  • PERSISTENT_VALIDATE_RUN_O3: 0 to skip the O3 lane
  • CARP_BIN: override the carp binary the script uses

Known compiler noise

The current Carp release sometimes emits warnings about internal generated Rc node handle types (No 'prn', Too many 'delete'). They're an artifact of how Rc.define interacts with the surrounding deftype system, not signals of a real problem. Treat them as expected noise as long as the suites and the sanitized lane are green.
